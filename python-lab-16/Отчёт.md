## Часть 1: GigaChat {#часть-1}

**Тема части:** применение ИИ-ассистента для генерации, рефакторинга, тестирования и документирования кода.  
**Инструменты:** GigaChat API, `gigachat`, `langchain-gigachat`, `.env`, Python.

### Цель

Освоить практическую работу с LLM через API и показать, как ИИ может помогать разработчику в типовых инженерных задачах.

### Архитектура решения

В основе решения используется класс `GigaChatAssistant`, который инкапсулирует:

- получение настроек из `.env`;
- инициализацию клиента GigaChat;
- построение prompt;
- получение текстового ответа модели;
- очистку результата от markdown-обёртки.

### Основные сценарии

#### 1. Генерация кода

Для генерации функции формируется prompt с ролью эксперта и ограничением формата ответа:

```python
prompt = f"""
Ты — эксперт по разработке на {language}. Напиши код для задачи:
{description}

Требования:
- аннотации типов
- docstring
- обработка ошибок

Верни только код.
"""
```

Результат используется как черновик, который затем проверяется разработчиком.

#### 2. Рефакторинг

В сценарии рефакторинга модели передаётся исходный код и требования к улучшению: читаемость, декомпозиция, типизация, сохранение логики.

#### 3. Генерация тестов

GigaChat применяется для подготовки тестов на `pytest` или `unittest`, включая:

- позитивные сценарии;
- негативные сценарии;
- граничные случаи.

#### 4. Генерация документации

Модель помогает быстро получить:

- docstring;
- README;
- API-документацию;
- пояснение к модулю.

### Практический вывод по части

GigaChat полезен как ускоритель работы, но не как автономный инженер. Основная ценность — в экономии времени на шаблонных задачах, черновиках и первом приближении решения.

## Часть 2: Airtable {#часть-2}

**Тема части:** разработка low-code CRM для управления заказами интернет-магазина.  
**Инструменты:** Airtable Base, Forms, Views, Lookup, Rollup, Formula, Automations.

### Цель

Собрать CRUD-систему без классического backend-кода и показать, как бизнес-данные и связи между сущностями можно настроить в low-code среде.

### Структура базы

В Airtable создаются таблицы:

- `Клиенты`
- `Товары`
- `Заказы`
- `Позиции заказа`

Связи строятся через `Link to another record`, что позволяет связать заказ с клиентом, а позицию заказа — и с заказом, и с товаром.

### Ключевые настройки

#### Таблица `Клиенты`

Содержит поля:

- авто-номер;
- ФИО;
- email;
- телефон;
- город;
- дата регистрации;
- статус;
- сумма покупок.

#### Таблица `Товары`

Описывает каталог товаров: название, категория, цена, остаток и описание.

#### Таблица `Заказы`

Содержит ссылку на клиента, дату, статус, способ оплаты, сумму и дату доставки.

#### Таблица `Позиции заказа`

Содержит связь на заказ и товар, количество, цену на момент заказа и формулу суммы позиции.

### Low-code механизмы

#### Lookup

Используется для автоматического подтягивания цены из таблицы товаров в позиции заказа.

#### Formula

Используется для вычисления суммы позиции заказа:

```text
{Количество} * {Цена на момент заказа}
```

#### Rollup

Используется:

- в заказе — для суммирования позиций;
- у клиента — для подсчёта общей суммы покупок.

#### Forms и Views

Созданные формы позволяют вводить заказы без ручного редактирования таблиц, а представления дают разные ракурсы работы:

- все заказы;
- активные заказы;
- заказы по клиентам;
- календарь доставок.

### Практический вывод по части

Airtable позволяет очень быстро собрать рабочую CRM-структуру и показать бизнес-логику без полноценной backend-разработки. Это особенно полезно для прототипов, внутренних инструментов и небольших процессов.

## Сравнение подходов {#сравнение}

| Критерий | GigaChat | Airtable |
|---|---|---|
| Что ускоряет | Написание кода | Создание приложения без кода |
| Основной пользователь | Разработчик | Аналитик, менеджер, citizen developer |
| Гибкость | Высокая, но требует кода | Быстрый старт, но меньше свободы |
| Главный риск | Ошибки модели | Ограниченность платформы |

Обе технологии направлены на ускорение разработки, но делают это на разных уровнях: GigaChat помогает инженеру писать код, Airtable уменьшает сам объём необходимого кода.

## Ответы на вопросы {#вопросы}

**Почему сгенерированный ИИ код нельзя принимать без ревью?**  
Потому что модель может ошибиться в логике, безопасности, типах данных и edge cases.

**Что делает prompt качественным?**  
Он задаёт чёткую задачу, формат выхода и ограничения, уменьшая вероятность неоднозначного ответа.

**В чём польза автоматической генерации тестов?**  
Она ускоряет старт тестового набора, но не заменяет ручной анализ бизнес-логики.

**Что такое lookup в Airtable?**  
Это механизм подтягивания поля из связанной таблицы.

**Что такое rollup в Airtable?**  
Это агрегирование значений из связанной таблицы: сумма, количество, среднее и другие операции.

**Почему Airtable считают low-code, а не no-code?**  
Потому что основной функционал собирается без программирования, но при интеграциях и сложной логике код всё же может понадобиться.
