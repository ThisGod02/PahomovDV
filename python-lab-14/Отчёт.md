## Часть 1: PostgreSQL {#часть-1}

**Тема части:** проектирование схемы интернет-магазина, SQL-аналитика, индексы и анализ плана выполнения.  
**Инструменты:** PostgreSQL, `psql`, SQL Editor, `EXPLAIN ANALYZE`.

### Цель

Получить навыки работы с реляционной СУБД PostgreSQL: создать нормализованную схему, заполнить её тестовыми данными, подготовить аналитические SQL-запросы и выполнить базовую оптимизацию через индекс.

### Структура решения

В рамках части используются 4 таблицы:

- `users` — клиенты магазина;
- `products` — каталог товаров;
- `orders` — шапка заказа;
- `order_items` — состав заказа.

Такое разделение позволяет не дублировать данные пользователя и товара в каждой строке заказа и поддерживать целостность через внешние ключи.

### Подготовленная схема

Ключевые элементы схемы:

```sql
CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    full_name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(user_id),
    order_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status VARCHAR(50) DEFAULT 'pending'
);
```

Дополнительно используются таблицы `products` и `order_items`, где `order_items` связывает заказ с товаром и хранит количество и цену на момент покупки.

### Аналитические запросы

#### 1. Список заказов с итоговой суммой

Запрос соединяет `orders`, `users` и `order_items`, после чего агрегирует сумму заказа:

```sql
SELECT
    o.order_id,
    u.full_name,
    o.order_date,
    o.status,
    SUM(oi.quantity * oi.unit_price) AS total_amount
FROM orders o
JOIN users u ON o.user_id = u.user_id
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, u.full_name, o.order_date, o.status
ORDER BY total_amount DESC;
```

#### 2. Выручка по категориям

Здесь используется `GROUP BY` и фильтрация агрегатов через `HAVING`:

```sql
SELECT
    p.category,
    SUM(oi.quantity) AS total_sold,
    SUM(oi.quantity * oi.unit_price) AS total_revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
GROUP BY p.category
HAVING SUM(oi.quantity * oi.unit_price) > 10000;
```

#### 3. Топ покупателей через CTE

```sql
WITH user_totals AS (
    SELECT
        u.user_id,
        u.full_name,
        SUM(oi.quantity * oi.unit_price) AS total_spent
    FROM users u
    JOIN orders o ON u.user_id = o.user_id
    JOIN order_items oi ON o.order_id = oi.order_id
    GROUP BY u.user_id, u.full_name
)
SELECT * FROM user_totals
ORDER BY total_spent DESC
LIMIT 3;
```

### Оптимизация запроса

Для ускорения выборки по идентификатору заказа подготовлен индекс:

```sql
CREATE INDEX idx_order_items_order_id ON order_items(order_id);
```

Проверка выполняется командой:

```sql
EXPLAIN ANALYZE
SELECT * FROM order_items WHERE order_id = 1;
```

До индекса в плане обычно наблюдается `Seq Scan`, после индекса — `Index Scan`, что снижает стоимость доступа к данным при частой фильтрации по `order_id`.

## Часть 2: MongoDB {#часть-2}

**Тема части:** моделирование интернет-магазина в документной базе данных, CRUD и aggregation pipeline.  
**Инструменты:** MongoDB Atlas или локальный `mongosh`, MongoDB Compass.

### Цель

Изучить подход NoSQL на примере MongoDB: создать коллекции и документы, выполнить CRUD-операции, построить aggregation pipeline и сравнить документную модель с реляционной.

### Структура модели

Используются коллекции:

- `users`
- `products`
- `orders`

В отличие от PostgreSQL, массив позиций заказа хранится прямо внутри документа `orders` в поле `items`.

Пример документа заказа:

```javascript
{
  _id: 1,
  user_id: 1,
  order_date: new Date(),
  status: "completed",
  items: [
    { product_id: 1, quantity: 1, price: 75000 },
    { product_id: 2, quantity: 2, price: 1500 }
  ]
}
```

### CRUD-операции

#### READ

Для чтения заказов пользователя с суммой используется pipeline с `$lookup`, `$unwind` и `$addFields`:

```javascript
db.orders.aggregate([
  {
    $lookup: {
      from: "users",
      localField: "user_id",
      foreignField: "_id",
      as: "user_info"
    }
  },
  { $unwind: "$user_info" },
  { $match: { "user_info.email": "alice@example.com" } },
  {
    $addFields: {
      total_amount: {
        $sum: {
          $map: {
            input: "$items",
            as: "item",
            in: { $multiply: ["$$item.quantity", "$$item.price"] }
          }
        }
      }
    }
  }
]);
```

#### UPDATE

Массовое обновление используется для присвоения скидки дорогим заказам:

```javascript
db.orders.updateMany(
  { total_amount: { $gt: 80000 } },
  { $set: { discount: 10 } }
);
```

#### DELETE

Удаление старых отменённых заказов:

```javascript
const thirtyDaysAgo = new Date();
thirtyDaysAgo.setDate(thirtyDaysAgo.getDate() - 30);

db.orders.deleteMany({
  status: "cancelled",
  order_date: { $lt: thirtyDaysAgo }
});
```

### Aggregation pipeline по категориям

Для отчёта по категориям используется:

```javascript
db.orders.aggregate([
  { $unwind: "$items" },
  {
    $lookup: {
      from: "products",
      localField: "items.product_id",
      foreignField: "_id",
      as: "product_info"
    }
  },
  { $unwind: "$product_info" },
  {
    $group: {
      _id: "$product_info.category",
      total_quantity: { $sum: "$items.quantity" },
      total_revenue: { $sum: { $multiply: ["$items.quantity", "$items.price"] } },
      avg_price: { $avg: "$items.price" }
    }
  },
  { $sort: { total_revenue: -1 } }
]);
```

Этот pipeline является документным аналогом SQL-агрегации по категориям.

## Сравнение подходов {#сравнение}

| Критерий | PostgreSQL | MongoDB |
|---|---|---|
| Хранение заказа | Шапка + отдельные позиции | Один документ с массивом `items` |
| Целостность | Первичные и внешние ключи | Гибкая схема, контроль на уровне приложения |
| Аналитика | SQL, CTE, HAVING | Aggregation pipeline |
| Изменяемость структуры | Ниже | Выше |
| Удобство чтения агрегата | Среднее | Высокое |

В PostgreSQL сильнее формализованы связи и ограничения. В MongoDB проще работать с объектом заказа как с единым документом.

## Ответы на вопросы {#вопросы}

**В чём разница между `WHERE` и `HAVING`?**  
`WHERE` фильтрует исходные строки до группировки, а `HAVING` фильтрует уже агрегированные группы.

**Почему в `GROUP BY` нужно перечислять неагрегированные столбцы?**  
Потому что СУБД должна однозначно понимать, по каким полям строится каждая группа результата.

**Что означают `Seq Scan` и `Index Scan`?**  
`Seq Scan` — полный проход по таблице. `Index Scan` — использование индексной структуры для более быстрого поиска.

**Почему в MongoDB позиции заказа вложены в документ?**  
Потому что заказ удобно рассматривать как агрегат: все связанные данные читаются одним документом.

**Что делает `$lookup`?**  
Это стадия агрегационного конвейера, которая подтягивает документы из другой коллекции по условию соответствия полей.
