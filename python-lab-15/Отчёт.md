## Часть 1: ETL на Python {#часть-1}

**Тема части:** построение ETL-пайплайна для анализа продаж интернет-магазина.  
**Инструменты:** Python, pandas, numpy, SQLite, SQLAlchemy, matplotlib, seaborn.

### Цель

Получить практические навыки по извлечению данных из CSV, очистке и трансформации набора данных, расчёту аналитических агрегатов, загрузке результата в SQLite и визуализации итогов.

### Архитектура решения

ETL-пайплайн реализован в классе `SalesETLPipeline` и включает пять этапов:

1. `extract()`
2. `transform()`
3. `aggregate()`
4. `load_to_sqlite()`
5. `visualize()`

### Этап Extract

На этапе `extract()` CSV-файл загружается через `pandas.read_csv`, после чего логируется его структура. Основная задача — безопасно получить исходный слой данных и сохранить его в `self.raw_data`.

Пример логики:

```python
try:
    self.raw_data = pd.read_csv(self.csv_path)
    logger.info(f"Загружено {len(self.raw_data)} строк")
except FileNotFoundError:
    logger.error(f"Файл {self.csv_path} не найден")
    raise
```

### Этап Transform

На этапе очистки выполняются:

- удаление дубликатов;
- обработка пропусков;
- фильтрация аномалий;
- приведение типов;
- расчёт `total_amount`;
- добавление поля `month_year`.

Ключевой смысл этапа — превратить “грязный” CSV в консистентный набор данных для аналитики.

Пример вычисления суммы строки:

```python
df["total_amount"] = df["quantity"] * df["price_per_unit"]
df["month_year"] = df["order_date"].dt.strftime("%Y-%m")
```

### Этап Aggregate

После очистки данные агрегируются по категории и месяцу:

```python
self.aggregated_data = df.groupby(["category", "month_year"]).agg({
    "quantity": "sum",
    "total_amount": "sum",
    "price_per_unit": "mean",
    "order_id": "nunique"
}).rename(columns={
    "quantity": "total_quantity",
    "total_amount": "total_revenue",
    "price_per_unit": "avg_price",
    "order_id": "order_count"
}).reset_index()
```

Результатом становится небольшая аналитическая витрина, пригодная для отчётности.

### Этап Load

Очищенные и агрегированные данные сохраняются в SQLite:

- `sales_cleaned`
- `sales_aggregated`

Это отделяет слой подготовки от слоя итоговой аналитики.

### Этап Visualize

Финальный этап строит графики:

- выручка по категориям;
- динамика продаж по месяцам;
- доля категорий в выручке.

Таким образом ETL-процесс завершает не просто загрузку, а полноценный аналитический цикл.

## Часть 2: Apache Kafka {#часть-2}

**Тема части:** событийная потоковая обработка заказов интернет-магазина.  
**Инструменты:** Docker Compose, Kafka, ZooKeeper, Kafka UI, Python `kafka-python`.

### Цель

Развернуть Kafka-кластер, создать топик заказов, реализовать producer и consumer, а затем организовать обработку событий о новых заказах в реальном времени.

### Архитектура решения

Стенд состоит из трёх сервисов:

- `zookeeper`
- `kafka`
- `kafka-ui`

Producer генерирует заказы и публикует их в топик `orders`, consumer читает этот поток и обновляет статистику.

### Producer

Producer создаёт объект `KafkaProducer`, сериализует сообщения в JSON и отправляет их в Kafka:

```python
self.producer = KafkaProducer(
    bootstrap_servers=self.bootstrap_servers,
    value_serializer=lambda v: json.dumps(v).encode("utf-8"),
    key_serializer=lambda k: k.encode("utf-8") if k else None
)
```

Заказ включает:

- идентификатор;
- timestamp;
- данные клиента;
- позиции заказа;
- сумму заказа;
- способ оплаты.

### Consumer

Consumer подписывается на топик `orders` и собирает агрегаты в оперативной статистике:

- `total_orders`
- `total_revenue`
- `orders_by_category`
- `orders_by_city`
- `recent_orders`

Фактически это простой потоковый аналитический сервис.

### Зачем нужен Kafka UI

`Kafka UI` позволяет визуально проверить:

- наличие брокера;
- список топиков;
- сообщения в топике;
- состояние consumer groups.

Это делает лабораторную наглядной и упрощает отладку.

### Результат части

В результате формируется рабочая модель событийной архитектуры:

- producer публикует событие заказа;
- Kafka гарантирует доставку в топик;
- consumer независимо читает и обрабатывает событие;
- статистика обновляется в реальном времени.

## Сравнение batch и stream {#сравнение}

| Критерий | ETL | Kafka |
|---|---|---|
| Источник | CSV и выгрузки | События |
| Режим | Пакетный | Потоковый |
| Время реакции | Отложенное | Почти мгновенное |
| Тип результата | Очищенный слой и витрина | Онлайн-агрегаты |
| Основной сценарий | Аналитика и отчётность | Реакция на события |

ETL нужен, когда важна подготовка исторических данных и построение витрин. Kafka нужна, когда данные должны обрабатываться по мере возникновения.

## Ответы на вопросы {#вопросы}

**Почему ETL включает этап Transform?**  
Потому что сырые данные обычно содержат пропуски, дубликаты и аномалии, которые делают аналитику недостоверной.

**Зачем в ETL создаётся колонка `total_amount`?**  
Она переводит атомарные поля `quantity` и `price_per_unit` в бизнес-показатель, необходимый для отчётов.

**Почему очищенные данные и агрегаты сохраняются в разные таблицы?**  
Чтобы разделять слой подготовленных данных и слой итоговой аналитики.

**Что такое topic в Kafka?**  
Это логический поток сообщений, в который producer публикует события, а consumer их читает.

**Что такое offset?**  
Это позиция сообщения в partition, по которой consumer понимает, какие события уже обработаны.

**Чем потоковая обработка отличается от пакетной?**  
Пакетная обрабатывает накопленный набор данных целиком, а потоковая реагирует на каждое событие по мере поступления.
