### Ответы на вопросы

1. **Преимущество собственного движка Flutter:** Flutter не зависит от нативных виджетов — любой пиксель рисуется движком Skia/Impeller напрямую. Это даёт пиксельно-одинаковый результат на Android и iOS, позволяет делать кастомные анимации (CustomPainter) без ограничений нативного UI.
2. **60+ FPS в Flutter:** Рендеринг выполняется в отдельном потоке (Raster thread), UI-поток только выдаёт описания сцены (display list). `AnimationController` использует `vsync` с `TickerProvider`, чтобы синхронизироваться с VSYNC дисплея.
3. **Датчики в Flutter vs нативная разработка:** В нативе — прямой доступ к SensorManager (Android) или CMMotionManager (iOS). В Flutter — через плагины (sensors_plus, pedometer), которые оборачивают нативный API в Dart-стримы.
4. **Паттерны управления состоянием:** Использован Provider (`ChangeNotifier`). Глобальное состояние (список тренировок, датчики) — в провайдерах. Локальное (текущий экран) — `setState`. Provider выбран как простое и официально рекомендованное решение.

### Ответы на вопросы

1. **Преимущества KMM перед Flutter/RN:** KMM разделяет только бизнес-логику, UI остаётся полностью нативным. Нет «мостов» — Kotlin-код компилируется в нативный binary для каждой платформы.
2. **expect/actual:** Компилятор требует `actual`-реализацию для каждого `expect`-объявления на каждой целевой платформе. Это гарантирует, что платформо-зависимый код присутствует на этапе компиляции.
3. **Сложности интеграции с iOS:** Нужно собрать XCFramework (`./gradlew shared:assembleSharedXCFramework`), правильно настроить Podfile/Xcode-пути. Coroutines на iOS требуют оборачивания в `ObservableObject`.
4. **Потокобезопасность:** SQLDelight работает с `Dispatchers.IO` через `asFlow().mapToList(Dispatchers.IO)`. ViewModel использует `SupervisorJob + Dispatchers.Main`, что изолирует ошибки дочерних корутин.

function cleanOldCache(maxAge = 30 * 24 * 60 * 60 * 1000) {
    const req = indexedDB.open('NewsDB', 1);
    req.onsuccess = e => {
        const db = e.target.result;
        const tx = db.transaction(['savedArticles'], 'readwrite');
        const index = tx.objectStore('savedArticles').index('savedAt');
        index.openCursor(IDBKeyRange.upperBound(Date.now() - maxAge)).onsuccess = ev => {

**E. Стратегия Network-First с таймаутом (service-worker.js)**

```javascript
async function networkFirst(request, timeout = 3000) {
    const timeoutRes = new Promise(resolve =>
        setTimeout(async () =>
            resolve(await caches.match(request) || new Response('Timeout', {status:408})), timeout)
    );
    const fetchRes = fetch(request.clone())
        .then(async res => {
            if (res.ok) (await caches.open(API_CACHE)).put(request, res.clone());
            return res;
        })
        .catch(async () => await caches.match(request) || new Response('Offline', {status:503}));
    return Promise.race([fetchRes, timeoutRes]);
}
```

### Ответы на вопросы

1. **Cache-First vs Network-First:** Cache-First — сначала читаем кэш, идеально для статики (CSS, иконки). Network-First — сначала сеть, резерв — кэш, для динамичных данных (API новостей). Добавление таймаута делает Network-First устойчивым к медленному соединению.
2. **Service Worker в офлайн-режиме:** SW перехватывает все `fetch`-запросы браузера. При отсутствии сети возвращает ответ из Cache Storage или IndexedDB, созданный при прошлом онлайн-сеансе.
3. **Параметры manifest.json:** `display: standalone` — убирает браузерный chrome; `start_url` — точка запуска при открытии из иконки; `icons` — набор для разных DPI; `theme_color` — цвет статус-бара; `shortcuts` — быстрые действия из лонг-тапа иконки.
4. **Push в PWA vs нативные:** Push PWA требует разрешения браузера и VAPID-ключей; сообщение идёт через сервер пуш-провайдера (FCM/APNS). Нативные пуши — через Platform SDK (FCM/APNS напрямую). В PWA не нужен apk/ipa — достаточно браузера, но iOS до Safari 16.4 не поддерживал PWA Push.
5. **Ограничения PWA:** Нет доступа к Bluetooth/NFC (кроме Chrome), ограниченный фоновый режим, нет App Store листинга (кроме PWA Builder), на iOS — ограниченное хранилище (~50 МБ), нет поддержки фоновых уведомлений без браузера.

## Общий вывод {#вывод}

В ходе лабораторной работы №12 изучены пять современных подходов к разработке мобильных и веб-приложений:

- **Jetpack Compose** — лучший выбор для чисто Android-приложений: максимальная интеграция с Google-экосистемой, нативная производительность, декларативный Kotlin-синтаксис.
- **React Native** — наиболее популярная кроссплатформенная технология благодаря огромной экосистеме npm и знакомому React-синтаксису; компромисс между нативностью и скоростью разработки.
- **Flutter** — сильная сторона — единый высококачественный UI на всех платформах через собственный движок; выделяется богатыми анимационными возможностями (CustomPainter).
- **KMM** — уникальный подход: переиспользование только бизнес-логики, UI остаётся 100% нативным. Сложная настройка, но максимальная нативность.
- **PWA** — «бесплатное» мобильное приложение из веб-сайта: установка без магазинов, офлайн-режим, push-уведомления. Идеально для контентных сервисов с ограниченным бюджетом.

Выбор технологии определяется задачей: нужен максимум нативности → Jetpack Compose/KMM; нужна быстрая кроссплатформа с нативным UI → React Native; нужны кастомные анимации на всех платформах → Flutter; нужен охват без магазинов приложений → PWA.
