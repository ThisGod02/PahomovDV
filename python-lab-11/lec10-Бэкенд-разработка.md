### **Слайд 10: REST API vs GraphQL**

- **REST (Representational State Transfer):**
  - Ресурсо-ориентированный
  - Стандартные HTTP-методы (GET, POST, PUT, DELETE)
  - Фиксированные эндпоинты
  - Over-fetching/under-fetching проблема
- **GraphQL:**
  - Запросо-ориентированный
  - Один эндпоинт для всех операций
  - Клиент определяет, какие данные нужны
  - Сильная типизация схемы

**Заметки:**
GraphQL решает проблемы REST, но добавляет сложность. Выбор зависит от специфики проекта.

### **Слайд 11: GraphQL: пример схемы и запроса**

```graphql
# Схема GraphQL
type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  content: String!

  users: [User!]!

    name
    posts {
      title
    }
  }
```

**Заметки:**
GraphQL особенно полезен для мобильных приложений, где важно минимизировать трафик.

### **Слайд 17: Пример REST API на Express (Node.js)**

```javascript
const express = require('express');
const app = express();
app.use(express.json());

// Моковые данные
let users = [
  { id: 1, name: 'Анна', email: 'anna@example.com' },
  { id: 2, name: 'Иван', email: 'ivan@example.com' }
];

// GET /users - получить всех пользователей
app.get('/users', (req, res) => {
  res.json(users);
});

// GET /users/:id - получить пользователя по ID
app.get('/users/:id', (req, res) => {
  const user = users.find(u => u.id === parseInt(req.params.id));
  if (!user) return res.status(404).json({ error: 'Пользователь не найден' });
  res.json(user);
});

// POST /users - создать нового пользователя
app.post('/users', (req, res) => {
  const newUser = {
    id: users.length + 1,
    name: req.body.name,
    email: req.body.email
  };
  users.push(newUser);
  res.status(201).json(newUser);
});

app.listen(3000, () => console.log('Сервер запущен на порту 3000'));
```

**Заметки:**
Express позволяет быстро создавать REST API. Важно правильно обрабатывать ошибки и валидировать данные.

### **Слайд 18: Пример REST API на FastAPI (Python)**

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List

app = FastAPI()

# Pydantic модель для валидации
class User(BaseModel):
    name: str
    email: str

# Моковые данные
users_db = []

# GET /users
@app.get("/users", response_model=List[User])
async def get_users():
    return users_db

# GET /users/{user_id}
@app.get("/users/{user_id}")
async def get_user(user_id: int):
    if user_id >= len(users_db):
        raise HTTPException(status_code=404, detail="Пользователь не найден")
    return users_db[user_id]

# POST /users
@app.post("/users", response_model=User)
async def create_user(user: User):
    users_db.append(user)
    return user

# Автоматическая документация доступна по:
# http://localhost:8000/docs (Swagger UI)
# http://localhost:8000/redoc (ReDoc)
```

**Заметки:**
FastAPI автоматически генерирует документацию и валидирует данные через Pydantic. Асинхронная обработка запросов.

### **Слайд 19: Аутентификация и авторизация**

- **Аутентификация:** Кто вы? (проверка личности)
  - Логин/пароль
  - OAuth 2.0 (через Google, GitHub и др.)
  - JWT (JSON Web Tokens)
  - Биометрия
- **Авторизация:** Что вам разрешено? (проверка прав)
  - RBAC (Role-Based Access Control)
  - ABAC (Attribute-Based Access Control)
  - ACL (Access Control Lists)

```javascript
// Пример JWT middleware в Express
const jwt = require('jsonwebtoken');

const authenticate = (req, res, next) => {
  const token = req.header('Authorization')?.replace('Bearer ', '');
  if (!token) return res.status(401).json({ error: 'Требуется аутентификация' });
  
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;
    next();
  } catch (err) {
    res.status(401).json({ error: 'Неверный токен' });
  }
};

// Использование
app.get('/profile', authenticate, (req, res) => {
  res.json({ user: req.user });
});
```

**Заметки:**
JWT — популярное решение, но важно хранить секретный ключ в безопасности и устанавливать разумное время жизни токена.

### **Слайд 20: OAuth 2.0 и OpenID Connect**

- **OAuth 2.0:** Протокол авторизации (делегирование прав)
- **OpenID Connect (OIDC):** Надстройка над OAuth для аутентификации
- **Поток Authorization Code (наиболее безопасный):**
  1. Клиент перенаправляет пользователя к провайдеру
  2. Пользователь аутентифицируется и дает согласие
  3. Провайдер перенаправляет обратно с code
  4. Клиент обменивает code на access token
- **Использование:** "Войти через Google/Facebook/GitHub"

**Заметки:**
Не путайте OAuth (авторизация) и OIDC (аутентификация). Всегда используйте PKCE для нативных и SPA приложений.

### **Слайд 22: Кэширование для повышения производительности**

- **Зачем кэшировать?**
  - Уменьшение нагрузки на БД
  - Ускорение ответов
  - Снижение задержки
- **Уровни кэширования:**
  - **Клиентский:** браузер, CDN
  - **Серверный:** Redis, Memcached
  - **БД:** query cache, materialized views
- **Стратегии:**
  - **Cache-Aside (Lazy Loading):** Проверять кэш, если нет — загружать из БД
  - **Write-Through:** Записывать и в кэш, и в БД одновременно
  - **Write-Behind:** Сначала в кэш, потом асинхронно в БД

**Заметки:**
Кэширование — палка о двух концах. Неправильная инвалидация кэша может привести к устаревшим данным.

### **Слайд 23: Пример кэширования с Redis**

```javascript
const express = require('express');
const redis = require('redis');
const app = express();

const client = redis.createClient();
await client.connect();

// Middleware для кэширования GET запросов
const cacheMiddleware = (req, res, next) => {
  const key = `cache:${req.originalUrl}`;
  
  client.get(key).then(cachedData => {
    if (cachedData) {
      return res.json(JSON.parse(cachedData));
    }
    
    // Сохраняем оригинальный метод res.json
    const originalJson = res.json;
    res.json = function(data) {
      // Кэшируем на 5 минут
      client.setEx(key, 300, JSON.stringify(data));
      originalJson.call(this, data);
    };
    
    next();
  });
};

// Использование
app.get('/api/products', cacheMiddleware, async (req, res) => {
  const products = await db.getProducts(); // Долгая операция
  res.json(products);
});
```

**Заметки:**
Redis — популярное in-memory хранилище для кэширования. Важно правильно устанавливать TTL (время жизни) для кэшированных данных.

### **Слайд 28: Rate Limiting: защита от перегрузки**

- **Зачем ограничивать запросы?**
  - Защита от DDoS-атак
  - Предотвращение злоупотреблений API
  - Обеспечение fair use
  - Защита внутренних ресурсов
- **Алгоритмы:**
  - **Fixed Window:** счетчик за фиксированный интервал
  - **Sliding Window:** более точный, но сложнее
  - **Token Bucket:** позволяет bursts
  - **Leaky Bucket:** сглаживает нагрузку

```javascript
// Пример rate limiting middleware для Express
const rateLimit = require('express-rate-limit');

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 минут
  max: 100, // максимум 100 запросов с одного IP
  message: {
    error: 'Слишком много запросов. Попробуйте позже.'
  },
  standardHeaders: true, // Возвращает информацию в заголовках
  legacyHeaders: false,
});

// Применяем ко всем запросам
app.use(limiter);
```

**Заметки:**
Rate limiting должен быть достаточно строгим для защиты, но не слишком строгим для легитимных пользователей.

### **Слайд 31: Производительность бэкенда: оптимизации**

- **Профилирование и метрики:**
  - Измерение времени ответа (P95, P99)
  - Анализ узких мест (bottlenecks)
  - Мониторинг использования ресурсов (CPU, память, I/O)
- **Оптимизации базы данных:**
  - Правильные индексы (индексировать часто искомые поля)
  - Оптимизация запросов (EXPLAIN ANALYZE в PostgreSQL)
  - Кэширование результатов сложных запросов
  - Использование read replicas для распределения нагрузки
- **Оптимизации кода:**
  - Асинхронная обработка там, где возможно
  - Пакетная обработка запросов (batch processing)
  - Ленивая загрузка данных (lazy loading)

**Пример оптимизации запроса:**
```sql
-- До оптимизации (медленно на больших таблицах)
SELECT * FROM orders WHERE DATE(created_at) = '2026-01-15';

-- После оптимизации (использует индекс по created_at)
SELECT * FROM orders 
WHERE created_at >= '2026-01-15 00:00:00' 
  AND created_at < '2026-01-16 00:00:00';
```

**Заметки:**
Производительность бэкенда критически важна для пользовательского опыта. Начинайте с измерения — нельзя оптимизировать то, что нельзя измерить. Используйте APM-инструменты (Application Performance Monitoring) для постоянного мониторинга. База данных часто становится узким местом — уделяйте особое внимание оптимизации запросов, правильно проектируйте индексы, но помните, что каждый индекс замедляет операции записи. Для сложных вычислений рассмотрите вынос логики в отдельные сервисы или использование специализированных БД (например, ClickHouse для аналитики).

### **Слайд 32: Безопасность бэкенд-приложений**

- **OWASP Top 10 для API:**
  1. Сломанная аутентификация
  2. Сломанная авторизация  
  3. Чрезмерное раскрытие данных
  4. Отсутствие лимитов ресурсов
  5. Сломанная аутентификация на уровне функций
  6. Массовое присвоение (mass assignment)
  7. Неправильная настройка безопасности
  8. Инъекции (SQL, NoSQL, команды)
  9. Неправильное управление активами
  10. Недостаточное логирование и мониторинг

- **Защитные меры:**
  - Валидация всех входных данных
  - Использование параметризованных запросов
  - Хеширование паролей с солью (bcrypt, Argon2)
  - HTTPS везде (HSTS, TLS 1.3)
  - Заголовки безопасности (CSP, X-Frame-Options)

```javascript
// Пример защиты от SQL-инъекций с параметризованными запросами
// ❌ ПЛОХО (уязвимо к SQL-инъекциям):
const query = `SELECT * FROM users WHERE email = '${email}'`;

// ✅ ХОРОШО (защищено):
const query = 'SELECT * FROM users WHERE email = $1';
const values = [email];
const result = await db.query(query, values);
```

**Заметки:**
Безопасность должна быть встроена в процесс разработки с самого начала (Security by Design). Регулярно проводите аудиты безопасности, используйте статические анализаторы кода (SAST) и динамические сканеры (DAST). Не доверяйте данным от клиента — валидируйте всё на стороне сервера. Храните секреты (API-ключи, пароли БД) в безопасных хранилищах (Vault, AWS Secrets Manager), а не в коде. Реализуйте принцип наименьших привилегий — каждый сервис должен иметь минимально необходимые права. Не забывайте про security headers и регулярно обновляйте зависимости для закрытия известных уязвимостей.

### **Слайд 33: Масштабирование бэкенда**

- **Вертикальное масштабирование (scale up):**
  - Увеличение мощности сервера (CPU, RAM, SSD)
  - Простота реализации
  - Ограниченный предел (один сервер)
  - Single point of failure
- **Горизонтальное масштабирование (scale out):**
  - Добавление большего количества серверов
  - Большая отказоустойчивость
  - Сложность координации
  - Требует stateless-архитектуры
- **Гибридный подход:**
  - Балансировщики нагрузки (Nginx, HAProxy)
  - Сессии в Redis или БД
  - Кэширование на всех уровнях
  - Базы данных с репликацией

**Архитектура масштабируемого приложения:**
```
Пользователи → Load Balancer → [App Server 1, App Server 2, ...] 
                             → [Redis Cluster] → [Database Master + Read Replicas]
                             → [Message Queue] → [Worker Servers]
```

**Заметки:**
Выбор стратегии масштабирования зависит от характера нагрузки и требований приложения. Для начала часто достаточно вертикального масштабирования, но на определённом этапе придётся переходить к горизонтальному. Ключевое требование для горизонтального масштабирования — stateless-архитектура: серверы не должны хранить состояние пользовательских сессий локально. Используйте внешние хранилища (Redis, база данных) для сессий. Балансировщики нагрузки распределяют трафик между серверами — они же могут выполнять SSL-терминацию, кэширование и компрессию. Для баз данных используйте репликацию (master для записи, read replicas для чтения), а при очень больших нагрузках — шардинг (разделение данных по разным серверам).

### **Слайд 34: Бессерверные вычисления (Serverless)**

- **Преимущества FaaS (Function as a Service):**
  - Нет управления инфраструктурой
  - Автоматическое масштабирование (от 0 до тысяч экземпляров)
  - Оплата только за время выполнения (per 100ms)
  - Высокая доступность по умолчанию
- **Ограничения:**
  - Холодный старт (cold start) — задержка при первом вызове
  - Ограничения времени выполнения (обычно до 15 минут)
  - Ограничения памяти и CPU
  - Vendor lock-in (привязка к провайдеру)
- **Идеальные сценарии использования:**
  - Обработка файлов (изображения, видео)
  - Cron-задачи и периодические задачи
  - Валидация и трансформация данных
  - Веб-хуки и обработчики событий

**Пример AWS Lambda функции:**
```javascript
exports.handler = async (event) => {
  const { name } = JSON.parse(event.body);
  
  const response = {
    statusCode: 200,
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ 
      message: `Привет, ${name || 'мир'}!`,
      timestamp: new Date().toISOString()
    })
  };
  
  return response;
};
```

**Заметки:**
Serverless архитектура радикально меняет подход к разработке и развертыванию приложений. Вместо долгоживущих серверов вы развертываете отдельные функции, которые запускаются только при поступлении запроса. Это позволяет значительно снизить затраты для приложений с переменной или непредсказуемой нагрузкой. Однако есть и сложности: холодный старт может создавать задержки до нескольких секунд (особенно для функций на Java или .NET), отладка и мониторинг сложнее, чем у традиционных приложений. Также важно проектировать функции как идемпотентные (многократное выполнение дает тот же результат) и учитывать ограничения провайдера. Для снижения vendor lock-in используйте фреймворки вроде Serverless Framework, которые абстрагируют от конкретного провайдера.

### **Слайд 35: Event-Driven Architecture (EDA)**

- **Основные компоненты:**
  - **Производители (Producers):** генерируют события
  - **Потребители (Consumers):** обрабатывают события
  - **Брокеры сообщений (Message Brokers):** доставляют события
- **Паттерны EDA:**
  - **Event Sourcing:** хранение состояния как последовательности событий
  - **CQRS (Command Query Responsibility Segregation):** разделение операций записи и чтения
  - **Saga Pattern:** управление распределенными транзакциями через события
- **Преимущества:**
  - Слабая связность компонентов
  - Высокая масштабируемость
  - Устойчивость к отказам
  - Легкий аудит изменений

**Пример Saga Pattern для заказа:**
```
1. Order Service: Создает заказ → Отправляет OrderCreated
2. Payment Service: Получает событие → Спишет деньги → Отправляет PaymentProcessed
3. Inventory Service: Получает событие → Резервирует товар → Отправляет InventoryReserved
4. Shipping Service: Получает событие → Создает доставку → Отправляет ShippingScheduled
```

**Заметки:**
Event-Driven Architecture (EDA) особенно хорошо подходит для микросервисов и сложных бизнес-процессов. Вместо прямых вызовов между сервисами используются события, что делает систему более устойчивой к отказам — если один сервис временно недоступен, события накапливаются в брокере и будут обработаны позже. Однако EDA добавляет сложности: необходимо тщательно проектировать схемы событий, обеспечивать их обратную совместимость, реализовывать механизмы повторной обработки и компенсирующих действий для откатов. Для простых приложений EDA может быть избыточной, но для сложных систем с множеством интеграций это мощный паттерн.

### **Слайд 36: gRPC vs REST для микросервисов**

| **Критерий** | **REST/HTTP JSON** | **gRPC** |
|--------------|-------------------|----------|
| **Протокол** | HTTP/1.1 или HTTP/2 | HTTP/2 |
| **Формат данных** | JSON (текст) | Protocol Buffers (бинарный) |
| **Производительность** | Средняя | Высокая (меньше трафика, быстрее парсинг) |
| **Типизация** | Слабая (нужна дополнительная валидация) | Строгая (генерируется из .proto) |
| **Поддержка потоков** | Через WebSockets или Server-Sent Events | Встроенная поддерка streaming |
| **Клиентские библиотеки** | Ручное написание или кодогенерация | Автоматическая генерация для многих языков |
| **Инструменты** | Много (Postman, curl, браузер) | Меньше (в основном специализированные) |

**Пример .proto файла для gRPC:**
```protobuf
syntax = "proto3";

package user;

service UserService {
  rpc GetUser (GetUserRequest) returns (User) {}
  rpc CreateUser (CreateUserRequest) returns (User) {}
  rpc StreamUsers (StreamUsersRequest) returns (stream User) {}
}

message GetUserRequest {
  string user_id = 1;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
}
```

**Заметки:**
Выбор между REST и gRPC зависит от конкретных требований проекта. REST проще для начала, имеет отличную инструментальную поддержку и хорошо работает в веб-контексте. gRPC предлагает лучшую производительность (особенно для внутренней коммуникации между микросервисами), строгую типизацию и встроенную поддержку потоков. Для публичных API часто выбирают REST из-за его универсальности и простоты использования с JavaScript. Для внутренних сервисов, особенно в гетерогенных средах (разные языки программирования), gRPC может быть предпочтительнее благодаря автоматической генерации клиентов. Можно использовать и гибридный подход: REST для внешних API, gRPC для внутренней коммуникации.

### **Слайд 37: Современные тренды: Edge Functions**

- **Что такое Edge Functions:**
  - Код, выполняемый на edge-серверах CDN
  - Ближе к пользователю (меньшая задержка)
  - Примеры: Cloudflare Workers, Vercel Edge Functions
- **Преимущества:**
  - Глобальная производительность (ближе к пользователям)
  - Автоматическое масштабирование
  - Безсерверная модель
  - Интеграция с CDN
- **Использование:**
  - Персонализация контента (A/B тестирование)
  - Валидация запросов
  - Гео-роутинг
  - Кэширование с логикой

**Пример Cloudflare Worker:**
```javascript
addEventListener('fetch', event => {
  event.respondWith(handleRequest(event.request));
});

async function handleRequest(request) {
  const country = request.cf.country;
  const userAgent = request.headers.get('User-Agent');
  
  // Персонализация на основе местоположения
  let greeting = 'Hello';
  if (country === 'RU') greeting = 'Привет';
  if (country === 'DE') greeting = 'Hallo';
  
  // Определение типа устройства
  const isMobile = userAgent.includes('Mobile');
  const deviceType = isMobile ? 'mobile' : 'desktop';
  
  return new Response(
    JSON.stringify({ greeting, deviceType, country }),
    { headers: { 'Content-Type': 'application/json' } }
  );
}
```

**Заметки:**
Edge Functions представляют собой следующую эволюцию serverless — выполнение кода не просто в облаке, а на граничных серверах по всему миру. Это позволяет обрабатывать запросы буквально в нескольких миллисекундах от пользователя. Особенно полезно для глобальных приложений, где пользователи разбросаны по разным континентам. Edge Functions идеально подходят для задач, требующих минимальной задержки: персонализация контента, гео-роутинг, проверка авторизации, агрегация данных из разных источников. Однако есть ограничения: меньше доступных библиотек, ограничения по времени выполнения (обычно до 50ms), необходимость писать код, работающий в изолированной среде. Не вся логика приложения должна выполняться на edge — только та часть, где критически важна минимальная задержка.

- **VK Cloud (бывший Mail.ru Cloud):**
  - IaaS, PaaS решения
  - Собственные дата-центры
  - Интеграция с экосистемой VK
  - Решения для гейминга и медиа
- **Selectel:**
  - Более 10 лет на рынке
  - Акцент на IaaS
  - Хостинг, CDN, облачные серверы
  - Собственная сеть дата-центров
- **Преимущества российских платформ:**
  - Локальное законодательство (152-ФЗ, хранение данных в РФ)
  - Русскоязычная поддержка
  - Локальные точки присутствия (меньшая задержка для российских пользователей)
  - Платежи в рублях

**Заметки:**
В связи с текущей геополитической ситуацией и ужесточением законодательства о хранении персональных данных российских граждан, российские облачные платформы становятся всё более актуальными. Yandex Cloud предлагает наиболее полный набор услуг, сравнимый с AWS или Google Cloud. VK Cloud (ранее Mail.ru Cloud Solutions) активно развивается и интегрирует облачные сервисы с экосистемой VK. Selectel — более традиционный хостинг-провайдер с акцентом на IaaS. При выборе платформы учитывайте не только технические характеристики, но и юридические аспекты: где физически находятся данные, как обеспечивается их защита, соответствие 152-ФЗ. Также важно оценить экосистему — наличие managed сервисов для баз данных, Kubernetes, serverless функций, которые значительно упрощают эксплуатацию.

  - Знание принципов Data Engineering (обработка больших данных)

**Заметки:**

Это инженер, который понимает весь жизненный цикл приложения: от проектирования API и выбора базы данных до развертывания в production и мониторинга. Ключевой тренд — размывание границ между разработкой и эксплуатацией (DevOps культура). Разработчик должен уметь не только написать код, но и упаковать его в Docker-контейнер, настроить CI/CD пайплайн, добавить метрики и алерты. Также важно понимать бизнес-логику — без этого невозможно проектировать эффективные API. Не менее важны soft skills: умение объяснить технические решения нетехническим коллегам, писать понятную документацию, конструктивно участвовать в code review. Учитесь постоянно — технологии меняются быстро, и то, что было актуально 2 года назад, сегодня может устареть.
