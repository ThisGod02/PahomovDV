### Исправления

#### A. SQL-инъекция → Параметризованный запрос

```python
# ❌ ДО: конкатенация строк
query = f"SELECT * FROM users WHERE id = {user_id}"
cursor.execute(query)

# ✅ ПОСЛЕ: параметризованный запрос + валидация типа
uid = validate_integer(user_id, 'id')
cursor = conn.execute(
    "SELECT id, username, email FROM users WHERE id = ?",
    (uid,)
)
```

#### B. Hardcoded Secret → Переменная окружения

```python
# ❌ ДО:
API_KEY = "demo_training_key_do_not_use"

# ✅ ПОСЛЕ:
API_KEY = os.environ.get('API_KEY')
if not API_KEY:
    raise ValueError("API_KEY environment variable must be set")
```

#### C. Command Injection → Allow-list

```python
# ❌ ДО: shell=True + любой ввод
result = subprocess.check_output(cmd, shell=True)

# ✅ ПОСЛЕ: allow-list + shell=False
ALLOWED_COMMANDS = ['date', 'whoami', 'uptime', 'hostname']
if cmd not in ALLOWED_COMMANDS:
    return jsonify({"error": "Command not allowed"}), 403
result = subprocess.check_output([cmd])  # shell=False
```

#### D. Flask debug=True

```python
# ❌ ДО:
app.run(debug=True, host='0.0.0.0', port=5000)

# ✅ ПОСЛЕ:
debug_mode = os.environ.get('FLASK_DEBUG', 'false').lower() == 'true'
app.run(debug=debug_mode, host='0.0.0.0', port=5001)
```

### Задание B: XSS-уязвимости

#### Тест уязвимого приложения

```bash
# Автоматическая демонстрация уязвимого и защищённого поведения
npm test

# Или ручной запуск по отдельности
npm run start:vulnerable   # http://localhost:3000
npm run start:secure       # http://localhost:3001
```

**Тестовые XSS-payload'ы:**
```
1. Поле comment: <script>alert('XSS Stored!')</script>
2. Поле username: <img src=x onerror="alert(document.cookie)">
3. API: GET /api/comments → innerHTML без экранирования
```

В уязвимом приложении тег `<script>` выполняется для ВСЕХ посетителей страницы (Stored XSS).

#### Защита в secure_app.js

**1. Санитизация на сервере:**
```javascript
const sanitizeHtml = (input) => input
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');

username = sanitizeHtml(username || 'Anonymous').substring(0, 50);
comment  = sanitizeHtml(comment  || '').substring(0, 500);
```

**2. Безопасный EJS-шаблон:**
```ejs
<%# ❌ Опасно — не экранирует %>
<%- comment.comment %>

<%# ✅ Безопасно — экранирует HTML %>
<%= comment.comment %>
```

**3. Content Security Policy:**
```javascript
res.setHeader('Content-Security-Policy',
    `default-src 'self'; script-src 'self' 'nonce-${nonce}';`
);
```

**4. Безопасный JavaScript (textContent вместо innerHTML):**
```javascript
// ❌ Уязвимо
html += '<div>' + comment.comment + '</div>';
element.innerHTML = html;

// ✅ Безопасно
const div = document.createElement('div');
div.textContent = comment.comment;  // автоматически экранирует
container.appendChild(div);
```

## Сравнение инструментов {#сравнение}

| Характеристика | SAST (Bandit) | SCA (npm audit) | DAST (OWASP ZAP) |
|---|---|---|---|
| Что анализирует | Исходный код | Файлы зависимостей | Работающее приложение |
| Когда запускать | При каждом коммите | При установке пакетов | На staging-стенде |
| Скорость | Секунды | Секунды | Часы |
| Ложные срабатывания | Много | Мало | Средне |
| Находит | SQLi, XSS в коде | CVE в библиотеках | Конфигурационные проблемы |
| Код нужен | ✅ Да | Только manifest | ❌ Нет |

## Ответы на вопросы {#ответы}

**Q1: Какие уязвимости обнаружил Bandit?**  
B201 (Flask debug=True), B602 (subprocess shell=True), B608 (SQL строки), B105 (hardcoded secret).

**Q2: Почему параметризованные запросы защищают от SQLi?**  
БД разделяет структуру запроса и данные. Значение `1 OR 1=1` передаётся как строка и ищется буквально, никогда не интерпретируется как SQL-код.

**Q3: Разница shell=True vs shell=False?**  
`shell=True` запускает `/bin/sh -c "команда"` — позволяет `;`, `&&`, `|`. `shell=False` запускает программу напрямую без оболочки — пользовательский ввод не может изменить логику.

**Q4: Что SAST не обнаруживает?**  
Ошибки логики (IDOR), конфигурацию сервера, уязвимости в библиотеках (это SCA), уязвимости при взаимодействии компонентов (это DAST).

**Q5: В чём разница npm audit vs Snyk?**  
`npm audit` — только NPM Advisory Database, бесплатно. Snyk — несколько баз (NVD, GitHub Advisory, собственная), глубже анализирует транзитивные зависимости, предоставляет детальные инструкции по исправлению.

**Q6: Почему CSP эффективен против XSS?**  
Даже если XSS-скрипт внедрён в HTML, браузер проверяет его источник по политике. Без nonce или разрешённого источника — скрипт не выполнится. CSP — второй слой защиты независимо от экранирования.

## Общий вывод {#вывод}

Лабораторная работа показала практическое применение трёх классов инструментов DevSecOps:

- **SAST (Bandit)** находит уязвимости в коде на ранней стадии, но требует настройки для снижения ложных срабатываний.
- **SCA (npm audit)** критически важен: даже идеально написанный код уязвим, если зависимости устарели.
- **DAST (OWASP ZAP)** находит проблемы конфигурации, незаметные статическому анализу.

Принцип «defense in depth»: все три инструмента дополняют друг друга и должны использоваться вместе в CI/CD пайплайне.
