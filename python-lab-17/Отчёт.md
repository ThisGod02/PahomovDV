## Часть 1: Deploy в облаке {#часть-1}

**Тема части:** развёртывание fullstack-приложения с фронтендом на Vercel и бэкендом в Yandex Cloud.  
**Инструменты:** GitHub, Vercel, Docker, Yandex Cloud CLI, Yandex Container Registry, Managed PostgreSQL.

### Цель

Освоить базовый облачный деплой веб-приложения, разделив фронтенд и бэкенд по разным средам размещения.

### Архитектура решения

В рамках части рассматривается схема:

- фронтенд `Next.js` публикуется на Vercel;
- бэкенд `FastAPI` контейнеризуется в Docker;
- Docker image публикуется в Yandex Container Registry;
- бэкенд запускается в облачной среде;
- данные хранятся в PostgreSQL.

### Деплой фронтенда

Для фронтенда используется GitHub-репозиторий и автоматический импорт проекта в Vercel.  
Причины выбора:

- нативная поддержка Next.js;
- быстрый build;
- автоматический HTTPS;
- удобная работа с preview и production средой.

Контрольная команда локальной проверки:

```bash
npm run build
```

### Контейнеризация бэкенда

Для FastAPI подготавливается Dockerfile со slim-образом Python и установкой зависимостей через `requirements.txt`.

Ключевые преимущества контейнера:

- повторяемость окружения;
- переносимость;
- изоляция зависимостей;
- удобство публикации в registry.

Фрагмент запуска:

```dockerfile
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Работа с PostgreSQL

Бэкенд не должен хранить данные внутри контейнера, поэтому используется отдельный PostgreSQL-инстанс.  
Настройка доступа выполняется через переменные окружения:

```python
DATABASE_URL = os.getenv("DATABASE_URL")
POSTGRES_HOST = os.getenv("POSTGRES_HOST", "localhost")
POSTGRES_PORT = os.getenv("POSTGRES_PORT", "5432")
```

Такой подход делает конфигурацию переносимой между локальным и облачным окружением.

## Часть 2: CI/CD {#часть-2}

**Тема части:** автоматизация тестирования, сборки и деплоя через GitHub Actions.  
**Инструменты:** GitHub Actions, Vercel CLI, Docker Buildx, Yandex Container Registry, Bandit, pytest.

### Цель

Настроить pipeline, который автоматически проверяет качество кода и публикует новую версию приложения при изменении ветки `main`.

### Общая структура workflow

Пайплайн разделён на несколько job’ов:

1. `test-frontend`
2. `test-backend`
3. `deploy-frontend`
4. `build-and-push-backend`

Это разделение позволяет независимо валидировать части системы и запускать деплой только после успешных тестов.

### Проверки фронтенда

Для фронтенда выполняются:

- установка зависимостей через `npm ci`;
- lint;
- TypeScript type-check;
- тесты;
- production build.

Таким образом пайплайн проверяет как синтаксическое качество, так и возможность реальной сборки проекта.

### Проверки бэкенда

Для бэкенда workflow поднимает тестовый PostgreSQL-сервис и запускает:

- `pytest`;
- сбор покрытия;
- `bandit` для security scan.

Это особенно важно, потому что backend зависит от БД и требует не только статической, но и интеграционной проверки.

### Автоматический деплой фронтенда

После прохождения тестов фронтенд деплоится на Vercel через CLI:

```bash
vercel pull --yes --environment=production --token=$VERCEL_TOKEN
vercel build --prod --token=$VERCEL_TOKEN
vercel deploy --prebuilt --prod --token=$VERCEL_TOKEN
```

### Сборка и публикация backend image

Для бэкенда используется `docker/build-push-action`, который:

- собирает образ;
- тегирует его;
- публикует в Yandex Container Registry;
- использует кэш GitHub Actions для ускорения повторных сборок.

### Секреты и безопасная конфигурация

Все чувствительные данные выносятся в GitHub Secrets:

- `VERCEL_TOKEN`
- `VERCEL_ORG_ID`
- `VERCEL_PROJECT_ID`
- `YC_SA_KEY_JSON`
- `YC_REGISTRY_ID`
- `DB_URL`

Это предотвращает утечку токенов через репозиторий и позволяет безопасно использовать доступы внутри CI/CD.

## Сравнение ручного и автоматизированного деплоя {#сравнение}

| Критерий | Ручной деплой | CI/CD |
|---|---|---|
| Повторяемость | Зависит от человека | Формализована в workflow |
| Скорость релизов | Ниже | Выше |
| Контроль качества | Часто частичный | Встроенные проверки |
| Риск пропустить шаг | Высокий | Низкий |
| Наблюдаемость | Ограничена | Полные логи запусков |

CI/CD особенно полезен, когда проект состоит из нескольких частей и любое изменение должно проходить одинаковый путь проверки.

## Ответы на вопросы {#вопросы}

**Почему Vercel хорошо подходит для Next.js?**  
Потому что он нативно понимает build-модель Next.js и автоматически выдаёт production-ready deploy.

**Зачем бэкенд помещать в Docker?**  
Чтобы зафиксировать окружение и сделать развёртывание одинаковым в любой среде.

**Почему база данных вынесена отдельно от API-контейнера?**  
Потому что данные должны жить независимо от жизненного цикла контейнера приложения.

**Что делает GitHub Actions в этой лабораторной?**  
Автоматически запускает тесты, собирает приложения и выполняет деплой без ручных шагов.

**Зачем нужны GitHub Secrets?**  
Чтобы безопасно хранить токены, ключи и строки подключения, не помещая их в репозиторий.

**Что даёт IaC по сравнению с ручной настройкой облака?**  
Инфраструктура становится воспроизводимой, версионируемой и проверяемой так же, как код приложения.
